## North Star

demo-forge makes AI agents create demos. Not guides for humans to record demos — the agent reads the project, understands the hero feature, writes a script, records it, and produces publishable content.

The founder shouldn't be making GIFs. The agent should.

## What demo-forge Produces

| Format | Use Case | Tool Chain |
|--------|----------|------------|
| **Terminal GIF** | CLI tools, MCP servers running in terminal | asciinema → agg |
| **Terminal SVG** | Same, but crisper and smaller file size | asciinema → svg-term-cli |
| **Screenshot PNG** | Library output, API responses, config examples | Playwright or native screenshot |
| **Architecture diagram** | System design, data flow | Mermaid → SVG/PNG |
| **Comparison table image** | Feature comparison vs. alternatives | HTML → screenshot |
| **Social card** | Open Graph image for link previews | HTML template → PNG |

## What Makes This Different

1. **Agent-driven**: The agent reads the codebase and decides what to demo. No human scripting.
2. **Real output**: Demos run real code in real environments. No mocked terminal sessions.
3. **README-ready**: Output is sized, formatted, and linked for immediate README embedding.
4. **Brand-consistent**: Eidos AGI visual identity across all demo content.

## The Demo Pipeline

```
Read project → Identify hero feature → Write demo script → 
Test script → Record → Convert to target format → 
Optimize file size → Generate README embed markdown
```

## Integration with foss-forge

foss-forge's `/foss-check` audits whether a demo exists. foss-forge's `/foss-demo` assesses what kind of demo is needed and delegates here. demo-forge does the actual creation.

## Success Metric

Every Eidos AGI public repo has a demo in its README that was generated by demo-forge, not manually recorded by a human.
